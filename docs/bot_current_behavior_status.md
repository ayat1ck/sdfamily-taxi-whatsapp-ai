# Текущее поведение бота SD Family Taxi

Дата фиксации: 2026-06-17

Этот файл описывает, как бот работает сейчас по коду, без маркетинга и без предположений.

## 1. Общая схема ответа

Основная точка входа: `app/dialog/engine.py`, метод `handle_message(...)`.

На каждое входящее сообщение бот делает следующее:

1. Сохраняет сообщение в БД.
2. Определяет тип входящего сообщения:
   - `text`
   - `image`
   - `document`
   - `unsupported`
3. Смотрит текущее состояние водителя (`driver.state`).
4. До продолжения анкеты проверяет специальные команды и priority interrupts.
5. Только потом идет либо в регистрацию, либо в support-flow, либо в редактирование данных, либо в ответ менеджеру.

## 2. Какие ответы и сценарии уже реализованы

### 2.1. Priority routing до анкеты

Сейчас в `app/dialog/engine.py::_handle_priority_interrupts(...)` добавлена приоритетная маршрутизация.

До обычного шага регистрации бот пытается определить:

1. Запрос оператора / менеджера
2. Уже зарегистрированный водитель
3. Перевод в самозанятые
4. Проблемы входа в Яндекс Про
5. Проверка статуса заявки
6. Вопросы по тарифам
7. Запрос на изменение данных

### 2.2. Оператор / живой человек

Если сообщение похоже на:

- `оператор`
- `менеджер`
- `живой человек`
- `соедините`
- `поддержка`

бот:

- ставит `driver.requires_attention = true`
- переводит `driver.dialog_mode = manual`
- переводит заявку в `awaiting_manager_review`
- пишет system trace / event
- отвечает:

`Ваш запрос передан менеджеру. Ожидайте ответа.`

Регистрация дальше не продолжается.

### 2.3. Уже зарегистрированный водитель

Если бот понимает, что водитель уже подключен, он не должен запускать новую регистрацию.

Сейчас для этого есть ответ-меню:

1. Стать самозанятым
2. Изменить данные
3. Сменить автомобиль
4. Помощь со входом
5. Поддержка

### 2.4. Самозанятый

Если водитель пишет что-то вроде:

- `хочу стать самозанятым`
- `сделайте смз`
- `парковый самозанятый`

бот:

- не начинает новую анкету,
- переводит кейс в ручную обработку,
- ставит `awaiting_manager_review`,
- отвечает, что запрос передан менеджеру.

### 2.5. Вход в Яндекс Про

Если сообщение похоже на:

- `не могу войти`
- `не заходит`
- `ошибка входа`
- `нет парка`
- `не отображается таксопарк`

бот идет в support-flow по Яндекс Про, а если не может отработать сценарий автоматически, передает менеджеру.

### 2.6. Статус заявки

Если пользователь спрашивает:

- `когда будет готово`
- `где моя заявка`
- `не вижу парк`
- `не пришло приглашение`

бот пытается вернуть статус заявки.

Если нормальный статус-ответ не получается, кейс эскалируется менеджеру.

### 2.7. Тарифы

Если спрашивают:

- `включите комфорт`
- `включите межгород`
- `включите экспресс`
- `подключите тариф`
- `почему нет заказов`

бот не должен отвечать про комиссию парка.

Сейчас такие сообщения уводятся в ручную обработку менеджером.

### 2.8. Изменение данных

Если водитель пишет:

- `поменял машину`
- `сменил номер`
- `изменить данные`
- `изменить права`
- `обновить документы`

бот старается не перезапускать регистрацию, а переключать кейс на изменение данных.

## 3. Как проходит регистрация сейчас

### 3.1. Новый водитель

Для нового водителя регистрация идет через state machine.

Основные шаги:

1. ФИО
2. Контактный номер
3. Город
4. Адрес
5. ИИН
6. Дата рождения
7. Водительский стаж
8. Данные ВУ
9. Данные авто
10. Документы
11. Подтверждение анкеты
12. Отправка в Yandex

### 3.2. Документы

Сейчас поддерживаются:

- текст
- изображение
- документ

Для документов есть отдельная логика распознавания:

- водительское удостоверение
- удостоверение личности
- техпаспорт / СТС
- селфи с правами

Часть данных бот может вытащить из PDF / фото автоматически.

### 3.3. Подтверждение анкеты

Перед отправкой в Yandex бот собирает summary через `_build_confirmation(...)`.

В summary сейчас выводятся:

- ФИО
- фамилия / имя / отчество
- город
- адрес
- ИИН
- дата рождения
- стаж
- номер ВУ
- даты ВУ
- условие работы
- дата принятия
- слабослышащий водитель: `да / нет`
- марка / модель / год / госномер / цвет

### 4.2. Как это работает

Если пользователь пишет что-то вроде:

- `исправь телефон на +7707...`
- `измени город на Алматы`
- `поменяй ИИН на ...`
- `исправь авто на Mercedes-Benz S-Class`

бот:

1. пытается определить поле,
2. нормализует значение,
3. применяет значение в БД,
4. пересобирает summary подтверждения.

### 4.3. Ограничения по редактированию

Редактирование работает лучше всего именно на этапе:

- `confirm_data`
- `yandex_error`

В других состояниях некоторые запросы на изменение данных могут уйти в ручную обработку менеджером, а не в мгновенное исправление поля.

## 5. Дубликаты

### 5.1. Что проверяется

Перед созданием профиля есть проверки:

- по ИИН
- по госномеру

### 5.2. Что происходит при дубле

Сейчас при дубле бот переводит водителя в state:

- `duplicate_rejected`

и ставит:

- `application.status = duplicate_rejected`
- `driver.duplicate_flag = true`

### 5.3. Исправлено недавно

Для `duplicate_rejected` теперь собран единый ответ, чтобы бот не прыгал между:

- `Сейчас ожидается текстовый ответ`
- `Поддерживаются только текст...`
- старым текстом про повторную регистрацию

Сейчас там должен быть единый ответ с вариантами дальнейших действий.

### 5.4. Что еще остается слабым местом

Сценарий дубля еще не является полноценным mini-flow.

То есть:

- дубль определяется,
- бот останавливает повторную регистрацию,
- но дальнейший support по восстановлению / изменению / переводу в СМЗ пока еще не доведен до идеальной разговорной логики.

## 6. Насколько правильно сейчас проходит регистрация

### 6.1. Что можно считать рабочим

Сейчас рабочими можно считать:

- старт новой регистрации
- сбор базовых текстовых полей
- прием документов
- распознавание части документов
- подтверждение анкеты
- корректировки на этапе confirm
- базовые support-flow по Яндекс Про
- защита от дублей
- отправка нормализованных данных в Yandex

### 6.2. Что уже проходит правильно

В нормальном кейсе бот умеет:

1. принять входящее сообщение,
2. довести до анкеты,
3. собрать данные,
4. показать сводку,
5. принять исправления,
6. отправить заявку в Yandex.

### 6.3. Что еще не идеально

По текущему коду и по реальным кейсам из чатов слабые места такие:

1. **Смешение support и регистрации**
   - иногда бот все еще отвечает как будто пользователь продолжает шаг анкеты, хотя он задал отдельный вопрос.

2. **Неустойчивая обработка неочевидных формулировок**
   - короткие свободные фразы, жаргон, сокращения и непрямые вопросы не всегда корректно распознаются.

3. **Duplicate support еще сырой**
   - дубль уже не троит так сильно, но логика после дубля еще не полноценная.

4. **Field edit хорош, но не абсолютно везде**
   - основной edit-mode реализован, но поведение зависит от текущего state.

5. **Часть текстов и fallback-веток исторически были битыми**
   - часть уже поправлена, но этот участок надо дальше дочищать по живым чатам.

6. **Тарифные и менеджерские кейсы чаще уходят в эскалацию, чем в умный диалог**
   - это безопасно, но не всегда user-friendly.

## 9. Честный вывод

### Работает уже нормально

- обычная регистрация нового водителя,
- сбор анкеты,
- прием документов,
- подтверждение,
- отправка в Yandex,
- базовые исправления полей на confirm,
- базовые support-flow,
- защита от дублей.

### Не доведено до идеала

- понимание свободной человеческой речи на каждом шаге,
- спокойная обработка дублей как отдельного support-сценария,
- жесткое соблюдение нового приоритета интентов во всех edge-case сообщениях,
- единообразие fallback-ответов,
- полная предсказуемость при смешанных вопросах.

## 10. Практический ответ на вопрос “проходит ли регистрация правильно?”

Да, **базовая регистрация в текущем коде проходит и может быть доведена до отправки в Yandex**.

Но ответ “все идеально” сейчас был бы нечестным.

Правильнее так:

- основной регистрационный контур рабочий;
- менять ключевые блоки анкеты пользователь уже в основном может;
- support и нестандартные диалоги еще требуют дальнейшей стабилизации;
- duplicate / manager / tariff / status сценарии уже выделены, но еще не доведены до уровня полностью надежного conversational assistant.

## 11. Что логично чинить следующим

Следующие наиболее полезные доработки:

1. Полный deterministic fallback для неясных сообщений:
   - `Не могу точно определить проблему. Передаю вопрос менеджеру.`

2. Полноценный mini-flow для `duplicate_rejected`:
   - восстановление,
   - СМЗ,
   - смена авто,
   - изменение данных.

3. Более жесткое разделение:
   - регистрация,
   - support,
   - статус,
   - тарифы,
   - менеджер.

4. Дополнительная нормализация разговорных формулировок:
   - русские сокращения,
   - казахские частые слова,
   - смешанные фразы.

5. Прогон по реальным чатам и зачистка оставшихся неестественных ответов.
