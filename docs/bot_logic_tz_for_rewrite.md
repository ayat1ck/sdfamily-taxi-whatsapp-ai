# ТЗ на переписывание логики WhatsApp-бота SD Family Taxi

Дата фиксации: 2026-07-05

Этот документ описывает текущую логику работы бота и правила, которые нужно сохранить при переписывании. Здесь зафиксировано именно поведение системы, без списка ошибок и без предложений по улучшению.

## 1. Общая роль бота

Бот является backend-first workflow-системой для регистрации водителей таксопарка через WhatsApp.

Основные задачи:

1. принять входящее сообщение из WhatsApp;
2. определить, к какой ветке относится сообщение;
3. провести пользователя по анкете регистрации;
4. принять и обработать фото и документы;
5. сохранить все данные в базе;
6. отправить заявку в Yandex Fleet;
7. отдать часть данных в Google Drive и Google Sheets;
8. перевести часть обращений в support-flow или на менеджера.

## 2. Принципы работы

1. Источник истины по состоянию диалога хранится в БД.
2. Каждое сообщение сохраняется до принятия решения по ответу.
3. Бот работает как state machine.
4. Приоритет у backend-логики выше, чем у AI.
5. Текстовые, фото и документные сообщения обрабатываются по разным веткам.
6. Специальные сценарии могут перехватывать любой этап анкеты.
7. Регистрация, support, редактирование и пострегистрационные сценарии должны быть разделены логически.

## 3. Каналы входа

### 3.1 WhatsApp webhook

Бот получает входящие сообщения через WhatsApp Cloud API webhook.

Поддерживаются два метода:

- `GET /webhooks/whatsapp` для проверки webhook;
- `POST /webhooks/whatsapp` для приема сообщений.

### 3.2 Типы сообщений

Поддерживаются:

- `text`
- `image`
- `document`

Все остальные типы сообщений считаются неподдерживаемыми и должны получать отдельный ответ.

## 4. Обработка входящего сообщения

Для каждого сообщения выполняется такой порядок действий:

1. сохранить входящее сообщение в БД;
2. определить водителя по номеру WhatsApp;
3. получить или создать заявку;
4. обновить признаки активности диалога;
5. определить тип сообщения;
6. проверить специальные команды и приоритетные сценарии;
7. выбрать нужную ветку обработки;
8. сформировать ответ;
9. отправить ответ в WhatsApp;
10. сохранить исходящее сообщение и статус внешней операции.

## 5. Приоритет маршрутизации

Логика ответа должна идти в таком порядке:

1. проверка статуса `duplicate_rejected`;
2. проверка типа `unsupported`;
3. обработка `image` и `document`;
4. обработка специальных команд;
5. обработка priority interrupts;
6. обработка pending-веток;
7. support-flow для уже зарегистрированных водителей;
8. support-flow для Yandex-сценариев;
9. обычная регистрация;
10. FAQ или fallback.

## 7. Регистрационный поток

Регистрация идет пошагово.

### 7.1 Последовательность

1. ФИО.
2. Телефон.
3. Город.
4. Адрес.
5. ИИН.
6. Дата рождения.
7. Стаж вождения.
8. Данные автомобиля.
9. Данные водительского удостоверения.
10. Документы.
11. Подтверждение анкеты.
12. Отправка в Yandex.

### 7.2 Логика переходов

1. Следующий шаг определяется текущим `driver.state`.
2. Если часть данных уже заполнена, бот может перейти дальше.
3. Если из документа удалось извлечь значения, они подставляются в соответствующие поля.
4. Перед финальным подтверждением формируется summary.
5. После подтверждения система готовит отправку в Yandex.

## 8. Документы и фото

### 8.1 Что принимает бот

1. изображения;
2. документы;
3. текстовый ввод для ручного заполнения данных.

### 8.2 Что делает документная ветка

1. связывает файл с водителем и заявкой;
2. сохраняет метаданные файла;
3. при необходимости загружает файл во внешнее хранилище;
4. пытается распознать данные;
5. нормализует найденные значения;
6. записывает результаты в профиль водителя, авто или документы;
7. переводит диалог к следующему шагу анкеты.

### 8.3 Типы документов

Поддерживаются сценарии для:

- водительского удостоверения;
- удостоверения личности;
- техпаспорта / СТС;
- селфи с правами.

### 8.4 Принцип выбора типа документа

1. Если текущий state жестко указывает на документ, приоритет у ожидания этого документа.
2. Если state более общий, тип может быть определен по результату распознавания.
3. Для PDF и некоторых распознанных случаев допустимо заполнение нескольких документных слотов сразу.

## 9. Подтверждение анкеты

Перед отправкой в Yandex бот формирует summary.

В summary должны попадать:

- ФИО;
- город;
- адрес;
- ИИН;
- дата рождения;
- стаж;
- номер ВУ;
- даты ВУ;
- условия работы;
- дата принятия;
- признак слабослышащего водителя;
- марка, модель, год, госномер, цвет автомобиля.

На этапе подтверждения пользователь может:

1. подтвердить данные;
2. исправить отдельное поле;
3. вернуться к редактированию.

## 10. Исправление данных

Исправление отдельных полей должно работать как отдельная логическая ветка, а не как новый старт анкеты.

Поддерживаются изменения таких групп данных:

- телефон;
- город;
- ИИН;
- дата рождения;
- номер и даты водительского удостоверения;
- автомобиль;
- документы;
- часть рабочих данных.

Редактирование особенно важно в состояниях:

- `confirm_data`
- `yandex_error`

## 11. Проверка дублей

Перед созданием или отправкой профиля выполняются проверки:

1. по ИИН;
2. по госномеру.

Если найден дубль:

- бот переводит диалог в `duplicate_rejected`;
- заявка помечается соответствующим статусом;
- дальнейшая регистрация как обычная не продолжается.

## 12. Отправка в Yandex

### 12.1 Что отправляется

В Yandex передаются:

- данные водителя;
- данные автомобиля;
- данные документов;
- рабочие параметры и профильные поля.

### 12.2 Перед отправкой

1. выполняется валидация обязательных данных;
2. собирается payload;
3. устанавливается статус отправки;
4. ответ Yandex записывается в заявку;
5. при ошибке заявка переводится в `yandex_error`.

### 12.3 После ошибки

Если Yandex возвращает ошибку, бот:

1. сохраняет причину;
2. показывает пользователю понятный ответ;
3. предлагает исправить нужные поля;
4. позволяет повторить отправку.

## 13. Support-flow

Support-flow нужен для сценариев, которые не относятся к обычной анкете.

Основные темы:

- вход в Yandex Pro;
- SMS и коды;
- неактивный аккаунт;
- выход на линию;
- тарифы;
- изменение данных;
- запрос к менеджеру;
- статус заявки.

Если вопрос не удается уверенно обработать автоматически, он передается в ручную обработку.

## 14. Уже зарегистрированный водитель

Если водитель уже завершил регистрацию:

1. новая анкета не запускается;
2. приоритет получает support-flow;
3. пользователь может задать вопрос по Yandex Pro, статусу или изменениям данных;
4. при необходимости запрос передается менеджеру.

## 15. AI-слой

AI здесь работает как вспомогательный механизм, а не как единственный источник решения.

### 15.1 Приоритет

1. сначала deterministic backend;
2. потом FAQ-логика;
3. потом optional LLM;
4. затем fallback.

### 15.2 Что должен делать AI

1. распознавать intent;
2. помогать с FAQ;
3. поддерживать регистрацию, если backend это разрешил;
4. возвращать структурированный результат;
5. не менять state произвольно;
6. не выдумывать данные.

### 15.3 Где AI особенно уместен

- FAQ;
- краткие пояснения;
- уточнение неполных ответов;
- помощь в распознавании intent.

## 16. Данные и сущности

### 16.1 Driver

Хранит:

- номер WhatsApp;
- ФИО;
- персональные данные;
- state;
- счетчики сообщений;
- флаги внимания;
- признаки дубля;
- назначенного менеджера.

### 16.2 Vehicle

Хранит:

- марку;
- модель;
- год;
- госномер;
- цвет;
- дополнительные признаки.

### 16.3 Application

Хранит:

- общий статус заявки;
- статус отправки в Yandex;
- ответы Yandex;
- ошибки Yandex;
- связку с водителем, авто и документами.

### 16.4 Message

Хранит:

- входящие сообщения;
- исходящие сообщения;
- тип сообщения;
- raw payload;
- статус доставки.

### 16.5 Document

Хранит:

- тип файла;
- источник;
- метаданные;
- распознанные поля;
- связь с водителем и заявкой.

### 16.6 ConversationEvent

Служит журналом значимых событий диалога:

- начало онбординга;
- переходы между шагами;
- исправления;
- эскалации;
- ручные переводы.

### 16.7 IntegrationJob

Сюда пишутся внешние операции:

- отправка в Yandex;
- работа с Google;
- другие интеграции;
- статусы и результаты.

## 17. Google и внешние интеграции

### 17.1 Google Drive

Используется для хранения файлов и документов.

### 17.2 Google Sheets

Используется для синхронизации заявки в таблицу.

### 17.3 Yandex Fleet

Используется для создания или обновления профиля водителя и для передачи данных заявки.

## 18. Админская часть

Админка должна позволять:

1. смотреть чаты;
2. смотреть заявки;
3. смотреть документы;
4. видеть аудиторские записи;
5. видеть AI traces;
6. видеть integration jobs;
7. управлять ручными сценариями;
8. понимать, что ушло в Yandex.

## 19. Сохранение контекста

Состояние сохраняется на нескольких уровнях:

1. `driver.state`
2. `application.status`
3. отдельные флаги и счетчики
4. история сообщений
5. события диалога
6. интеграционные записи

Это нужно, чтобы после любого входящего сообщения можно было восстановить, на каком шаге находится диалог и почему был выбран именно этот ответ.

## 20. Что обязательно учитывать при переписывании

1. Приоритет special routes должен быть выше обычной анкеты.
2. Документы не должны попадать в обычный текстовый поток.
3. Подтверждение анкеты должно остаться отдельным этапом.
4. Исправление отдельных полей должно работать без перезапуска.
5. Проверка дублей должна оставаться отдельной логикой.
6. Отправка в Yandex должна происходить только после валидации.
7. Support-flow должен работать отдельно от регистрации.
8. Для зарегистрированных водителей нужен отдельный сценарий общения.
9. Все сообщения должны сохраняться.
10. Внешние операции должны логироваться.
11. AI должен оставаться вспомогательным слоем.

## 21. Краткая формула поведения

Бот принимает сообщение из WhatsApp, определяет его тип и приоритетную ветку, сохраняет контекст, ведет водителя по state machine регистрации, распознает документы, формирует summary, принимает исправления, проверяет дубли, отправляет данные в Yandex и после этого продолжает работать как support-бот для зарегистрированного водителя.
